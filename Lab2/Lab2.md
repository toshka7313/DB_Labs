# Лабораторна робота №2

## Перетворення ER-діаграми у реляційну схему PostgreSQL

### Метою даної лабораторної роботи є перетворення ER-діаграми предметної області «Медичні записи» у реляційну схему та її реалізація у СУБД PostgreSQL.

---

**Система дозволяє:**
* Вести базу даних лікарів та їх спеціалізацій.
* Зберігати персональні дані пацієнтів.
* Реєструвати прийоми пацієнтів у лікарів із фіксацією скарг та діагнозів.
* Виписувати рецепти на медикаменти за результатами конкретного прийому.

---

**Реляційна схема бази даних:**
* **Doctor** (doctor_id, full_name, specialization, contact_number)
* **Patient** (patient_id, full_name, birth_date, phone, email)
* **Appointment** (appointment_id, patient_id, doctor_id, date_time, complaints, diagnosis)
* **Prescription** (prescription_id, appointment_id, medication_name, dosage, instructions)

---

**Основні звʼязки між таблицями:**
* Один **Doctor** проводить багато **Appointment** (1:N).
* Один **Patient** може мати багато записів про візити у таблиці **Appointment** (1:N).
* Один **Appointment** може передбачати виписку багатьох **Prescription** (1:N).
* Зв'язок багато-до-багатьох (M:N) між лікарем та пацієнтом реалізовано через проміжну таблицю прийомів (**Appointment**).
* Усі звʼязки реалізовані через зовнішні ключі (FOREIGN KEY) з правилом каскадного видалення (`ON DELETE CASCADE`).

---

### Обмеження цілісності (Constraints)

Для забезпечення коректності даних та дотримання бізнес-правил використано наступні обмеження:
* **UNIQUE**: для номерів телефонів (`phone`) та електронних адрес (`email`) пацієнтів, щоб уникнути дублювання контактів.
* **CHECK**: для перевірки, що дата народження пацієнта не є в майбутньому (`birth_date <= CURRENT_DATE`).
* **DEFAULT**: для автоматичного встановлення дати та часу прийому на поточний момент (`CURRENT_TIMESTAMP`).

### Тестування

Для перевірки коректності створення схеми було виконано:
* Створення всіх таблиць у PostgreSQL (середовище розгорнуто через Docker-контейнер).
* Додавання 3 тестових записів у кожну таблицю.
* Перевірка обмежень цілісності:
* Спроба додавання пацієнта з дубльованим email (спрацювало обмеження `UNIQUE`).
* Спроба додавання дати народження з майбутнього (спрацювало обмеження `CHECK`).
* Виконання тестового запиту `SELECT * FROM Doctor;` для демонстрації наявності даних.
